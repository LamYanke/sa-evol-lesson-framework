项目简介：种群进化 + 多角色经验库的代码加速框架
本项目面向一个常见却难落地的目标：在保证语义等价的前提下，用 LLM 自动改写代码以提升运行时性能。它解决的不是“写出一个更快版本的一次性 Prompt”，而是一类更贴近工程现实的痛点：单次生成缺乏测量与记忆、成功经验难以沉淀为可复用的“思路”、失败容易被重复试错、同一种优化话术难以适配算法层与实现层。Framework 2.0 通过带测量的进化循环 + 结构化经验库 + 多角色“部落”改写 + 融合与退火选择，把这些痛点收口到一条可重复的流水线里。

核心痛点与应对方式可概括为：
其一，纯 LLM 优化往往不可靠且不可比较——本框架用 IncrementalProfiler 做冒烟校验与指令级步数近似，产出可比的 speedup，只有“跑得对且更快”才进入选择与沉淀环节。其二，泛泛的上下文容易跑偏或抄坏代码——父代改写时注入 ANCHOR（原始代码，约束必须保留的逻辑） 与 ANTI-PATTERN（最近一次失败片段，明示勿复用），把“等价性”写进 Prompt 而不是事后指望模型自觉。其三，单靠一条链路的试错遗忘快、检索粗——LessonBank 2.0 用 AST 指纹去重，结合 BM25（结构文本相似） 与 CFG 嵌入（长度/循环/分支）上的角色加权距离 做混合检索，并把“成功改写”经由 LLM 差量概括为一条战术话术（Delta Learning），沉淀的是可迁移的优化思路，而非简单堆仓库码。其四，同一套经验不一定适合所有人——检索时对 Architect / Engineer / Refiner 使用不同的 CFG 权重与“同行加成”，体现种群使用不同成功经验的设计理念。其五，单行精英容易早熟——外层采用模拟退火式接受：在温度下降过程中以一定概率保留略差个体，种群规模 capped（如至多 3），维持探索与利用的平衡。

核心逻辑流（是否“长链推理”、是否“多 Agent”）：
主循环在 SA_Evol_Lesson_Framework.run 中展开，本质是多轮（max_iterations）的“分析—进化—评估—沉淀—融合—选择”，而不是单条超长 CoT。每一轮内部会先对当前最优做一次热点侧的轻量分析，再对每个父代个体：由 TribeSelector 决定激活哪些部落（当前实现为激活全部三类）；每个 TribeAgent 从自身角色视角，经 lesson_bank.retrieve 拉取与当前代码结构相近、又与角色匹配的 lessons，然后执行一次 mutate：单轮 LLM 调用生成新个体——因此角色之间是并行分工，不是递归式长链推理；链的“长度”主要来自外层迭代代数与评估次数的累积。

多 Agent 协作体现在：三个固定角色的 TribeAgent（Architect 偏向复杂度与数据结构、Engineer 偏向热点与不破坏算法骨架、Refiner 偏向 Python 惯用法）在同一父代上产生不同候选；另有 MergerAgent 在至少两个有效候选时介入：以速度最快的个体为 Leader，把其余 Top 候选相对原始代码的 diff 当作“补丁意图”，再经一次 Lead Integrator 式 LLM 调用做加权融合，尝试把互补的小优化吸收进最强基底。经验侧则还有隐式角色：Lesson Learner 总结成功战术、Debugger 概括失败教训，与部落的改写调用共同构成工具化、分身的多调用协作，但没有显式的对话式多智能体协商协议或树状搜索式长推理。

小结： 本项目的价值在于把“LLM 写代码”变成可度量、可记忆、可分工、可融合的闭环；长链主要体现在多轮进化与多步评估，而非单轮内的逐步思维链；多 Agent 体现为多角色部落 + 融合代理 + 经验学习/调试调用的协作式流水线。若将 API 与评测接口换成真实基准与更大测试集，该框架可直接作为论文或实验中的种群级、经验驱动的自动性能优化基座使用。
